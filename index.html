the pipeline

Idea → live in ~a day

01
Idea → one-line brief

What it does, who it's for, and the one thing that must be correct. That last clause is the whole design.

02
Scaffold the shape

A dependency-light engine + a surface + tests from day one. The engine never imports the surface.

03
Build the engine first, test it hard

Pin the tricky cases. If the engine is right, the product is trustworthy.

04
Wrap it in surfaces

One engine answers from many places — chat tool, bot, web UI — the surface only renders what the engine computes.

05
Make it real software

Installable, containerized, CI on every push. Offline always works — no demo should need credentials.

06
Publish & document

Ship to GitHub; write the architecture + a build-vs-production spec so the gaps are owned, not hidden.

decision log

Tradeoffs, on the record

Why MCP over a bespoke API?
A model can lean on a tool that's provably correct instead of re-deriving logic in a prompt. The protocol is the contract.
Why one engine, many surfaces?
New front-ends become wiring, not rebuilds. Match Mode is the same scoring as the MCP — verified 1:1, so behavior can't drift.
Why adapters & repositories?
A new CRM or data source is a new file, not a rewrite. Integrations stay additive.
Why local Repos over cloud-sync?
Build artifacts don't belong in a sync folder. Code lives local; GitHub is the backup and the shop window.
Why synthetic data, always?
The pattern is the proof. No former-employer IP — the ideas travel, the data doesn't.
